# Design: v0.5 Slice 1 — read-hardening remainder (S1 + S3 + C3-pagination)
Status: draft · Date: 2026-06-26 · Slug: `v0.5-slice1-read-hardening` · Plan: docs/plans/v0.5-slice1-read-hardening.md
Issue: #19 · resolves #15 (S1), #17 (S3), #14-part (C3 pagination)

## Summary
Three independent hardening changes to the v0.4 one-shot live read path, on top of the
already-shipped C1 discovery tolerance: a deadline + signal cancellation on the CLI live
path (S1, in `cmd/krsm`), preferred-version-only listing (S3, in `internal/cluster`), and
paginated lists (C3-pagination, in `internal/cluster`). All preserve the existing
fail-closed behaviour and the read-only invariant; none touch `closure`/`scope`/goldens.
Implemented and reviewed in order S1 → S3 → C3, each behind its own tests.

## Interfaces / contracts

### S1 — CLI timeout + signal cancellation (`cmd/krsm/main.go`)
- New flag on the `check` FlagSet: `--timeout <duration>` (Go `time.Duration`, default
  `30s`). `--timeout 0` means **no deadline** (signal cancellation only).
- `liveOpts` gains `timeout time.Duration`.
- `runCheckLive(args, o, stdout, stderr)`: builds the request context as
  ```
  ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
  defer stop()
  if o.timeout > 0 { ctx, cancel = context.WithTimeout(ctx, o.timeout); defer cancel() }
  ```
  and passes that `ctx` to `ResolveKind`/`State` (already ctx-aware). No signature change
  to the reader.
- **Errors:** a deadline/cancellation surfaces from `State`/`ResolveKind` as a
  discovery/list error and is returned as the existing fail-closed
  `check: cluster read failed, denying fail-closed: …` (exit 1). No partial verdict.

### S3 — preferred-version listing (`internal/cluster/reader.go`)
```go
// serverPreferredResources lists one APIResourceList per resource at its SERVER-PREFERRED
// version, with the same partial-discovery tolerance as serverResources (C1): keep the
// resolved lists, fail closed only if a closure-relevant (built-in) group is among the
// failed set. Used to pick LIST targets so a multi-version resource is listed once.
func serverPreferredResources(disc discovery.DiscoveryInterface) ([]*metav1.APIResourceList, error)
```
- `Reader.State` change: `scopeFromLists` keeps consuming the **all-version**
  `serverResources(disc)` (so the per-version `Namespaced` flag stays complete);
  `selectTargets` consumes `serverPreferredResources(disc)` instead. No change to
  `selectTargets`/`scopeFromLists`/`supportsList` signatures.

### C3 — paginated list (`internal/cluster/reader.go`)
```go
// listAll lists every object of gvr READ-ONLY across pages, following the Continue token
// with a bounded page size. It is the only place State reads objects; it uses only
// Resource(...).List (no write verb). Any page error fails closed (returns the error).
func (r *Reader) listAll(ctx context.Context, gvr schema.GroupVersionResource) ([]unstructured.Unstructured, error)

const listPageLimit = 500 // kubectl's default page size
```
- `Reader.State` change: the per-target body becomes `items, err := r.listAll(ctx, t)`.

## Data model
No new persisted types. S1 adds `liveOpts.timeout time.Duration`. S3/C3 operate on the
existing `[]*metav1.APIResourceList` and `[]unstructured.Unstructured`. `ListOptions`
gains `Limit: listPageLimit` and `Continue: <token>` between pages.

## Behavior

### S1
- Happy path: `--timeout` unset → 30s deadline; reads complete well within it → unchanged.
- Deadline exceeded / SIGINT / SIGTERM mid-read → `ctx` is Done → the in-flight `List`
  (or a ctx-aware discovery call) returns a context error → `State` returns it → fail-closed
  exit 1 with the cluster-read-failed reason. Never a closure over a partial snapshot.
- `--timeout 0` → no `WithTimeout`; only signal cancellation can abort.

### S3
- Discovery returns HPA at `autoscaling/v1` and `autoscaling/v2`; `ServerPreferredResources`
  returns only the preferred (`/v2`). `selectTargets(preferred)` yields one HPA GVR →
  exactly one HPA `List`, at `v2`. `scopeFromLists(all)` still records the `Namespaced` flag
  for both versions, so namespace resolution is unaffected.
- Partial discovery during preferred lookup: same fail-open-on-aggregated / fail-closed-on-
  built-in rule as C1 (shared `closureRelevantFailures`).

### C3
- A kind with more objects than one page: `listAll` issues `List{Limit:500}`, gets a
  `Continue` token + first page, re-lists with `{Limit:500, Continue:token}`, repeats until
  `Continue == ""`, returns the concatenation. Mid-pagination error (any page) → fail closed.
- A kind with ≤500 objects: one `List`, empty `Continue`, returns directly (unchanged shape).

## Components & responsibilities
- `cmd/krsm/main.go` — `--timeout` flag, `liveOpts.timeout`, the `signal.NotifyContext` +
  `WithTimeout` context in `runCheckLive` (S1). `main` exit mapping unchanged.
- `internal/cluster/reader.go` — `serverPreferredResources` (S3), `listAll` +
  `listPageLimit` (C3), and the two-line `State` rewiring. Reuses C1's
  `closureRelevantFailures`. Stays read-only.
- Tests: `cmd/krsm/*_test.go` (S1, via the existing `newLiveReader` injection seam),
  `internal/cluster/reader_test.go` + a new `read_hardening_test.go` (S3/C3). The shared
  test discovery helper defaults `ServerPreferredResources` to the full lists.

## Alternatives considered
- **S1 in `main` instead of `runCheckLive`.** Rejected: `runCheckLive` is the testable seam
  (hermetic via `newLiveReader`); keeping `main` thin keeps signal+timeout under test.
- **S3 by post-filtering all-version lists to "preferred" ourselves.** Rejected: the server
  already computes preferred version (`ServerPreferredResources`); re-deriving it risks
  diverging from the API server's choice. We only fall back to all-version lists for the
  scope (namespaced) map, which genuinely needs every version.
- **C3 unbounded list kept, rely on informers later.** Rejected: the one-shot CLI path is a
  real user path now; a single unbounded list is a present latency/OOM hazard. Pagination is
  cheap and local.
- **Switch listing to `ServerPreferredResources` AND drop the all-version scope read.**
  Rejected: scope resolution needs the `Namespaced` flag for every version that may appear
  on an object's `apiVersion`; preferred-only could miss a non-preferred version's flag.

## Trade-offs & risks
- Two discovery calls in `State` now (all-version for scope + preferred for targets). Cheap,
  cached server-side; acceptable for the one-shot path.
- Test doubles: `FakeDiscovery.ServerPreferredResources` returns `(nil,nil)` → the shared
  test discovery must serve preferred lists (default = the full lists) or existing reader
  tests would list nothing; the multi-version S3 test overrides preferred explicitly. The
  dynamic fake ignores `Limit/Continue`, so the C3 test uses a `list` reactor that returns a
  `Continue` token on page 1 and the remainder on page 2.
- Read-only invariant: `listAll` uses only `Resource(...).List`; `TestReaderUsesOnlyReadVerbs`
  and `TestPackageInvokesNoWriteVerbs` must stay green.

## Testability
- **S1:** `runCheckLive` with `liveOpts{timeout: 1*time.Nanosecond}` + a fake `liveReader`
  whose `State` blocks on `ctx.Done()` → returns a fail-closed error; assert exit-1 mapping
  and no report written. A `--timeout` parse test (duration + `0`).
- **S3:** a discovery double serving HPA at v1+v2 (all) and v2 (preferred) → `State` issues
  exactly one HPA `List` at v2 (inspect the dynamic fake's actions). Existing reader tests
  stay green via the default preferred=all.
- **C3:** a dynamic `list` reactor paging on `Continue` → `State` assembles all items across
  ≥2 `list` actions; a page-2 error → `State` fails closed.
- Regression: full `make check` (incl. read-only guards, corpus parity untouched).

## Open questions
None blocking — page size 500, default timeout 30s, and `--timeout 0 = no deadline` are
approved (plan).
