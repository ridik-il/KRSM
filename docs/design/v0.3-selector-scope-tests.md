# Test list: v0.3-selector-scope  (source: docs/design/v0.3-selector-scope.md)

Ordered as vertical slices — implement one before writing the next (red → green → refactor).
Every entry is observed through a **public interface**: `closure.Safe`'s returned `Decision`
(`Verdict` / `Escaping` / `Closure`), the `internal/scenario` loader + golden assertions, or the
`krsm check` CLI. No test touches `matchScope` directly (it stays unexported).

> **Build order note (for `tdd`):** the first green step is the *API rename* —
> `ScopeRef` → `ScopeClause{Dim, GVK, Namespace, Name, Selector}`, retype `Safe`/`matchScope`/
> `Scenario.Scope`/`parseScope`/`cmd/krsm` formatters, with the resource-dim path behaviourally
> identical. Tests 1–2 plus the **19 existing goldens** are the guard that the rename changed
> nothing. Only then add the `DimSelector` behaviour (3–7), then loader + scenario + CLI (8–11).

## Slice 1 — selector matching in the engine (`package closure`, no loader)
Built from `NewScanState(objs)` + `ScopeClause` values + `Safe`; observed via `Decision`.

1. [ ] A `resource`-dim clause admits a closure member it names exactly (member not in `Escaping`) — back-compat.
2. [ ] A `resource`-dim clause admits by `path.Match` glob, and a clause with an empty `Dim` is treated as `resource` — back-compat for dim-less scopes.
3. [ ] A `selector`-dim clause with `matchLabels` admits a member whose labels match; a same-kind/same-namespace member whose labels don't match escapes.
4. [ ] A `selector`-dim clause with `matchExpressions` admits/excludes correctly across `In`, `NotIn`, `Exists`, `DoesNotExist`, including the absence-sensitive cases (key missing).
5. [ ] A `selector`-dim clause does **not** match a candidate of a different `GVK.Kind` or different `Namespace`, even when its labels satisfy the selector (gating preserved).
6. [ ] An empty/nil `selector`-dim clause matches **nothing** — its candidate members escape (fail-safe; no silent namespace-wide over-grant).
7. [ ] A `selector`-dim clause does **not** match a candidate absent from `State` (labels unresolvable) — it escapes (fail-closed, DESIGN §5).

## Slice 2 — loader, golden scenario, CLI
8. [ ] **Regression:** the 19 existing dim-less `scope.yaml` files load as `resource` clauses and every existing golden scenario still produces its expected `Decision` (byte-for-byte verdict/closure/escaping/external).
9. [ ] The loader parses a `scope.yaml` clause with `dim: selector` + `matchExpressions` (and/or `matchLabels`) into a `selector` `ScopeClause` carrying the populated `closure.LabelSelector`.
10. [ ] **The proof:** golden scenario `20-scope-selector-precision` — `Safe` over the loaded scenario returns `Allow` with closure `{Deployment/prod/frontend, Pod/prod/frontend-aaa, Pod/prod/worker-bbb}` and empty `Escaping` (the same inputs Block when the selector clause is treated as resource-only).
11. [ ] `krsm check closure/testdata/scenarios/20-scope-selector-precision` prints the `selector` clause in the SCOPE line, reports `ALLOW`, and exits `0`; a `resource` clause still renders as before.
