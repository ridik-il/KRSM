# Design: v0.5 Slice 2 — shared per-object projection
Status: draft · Date: 2026-06-26 · Slug: `v0.5-slice2-shared-projection`
Issue: #20 · Plan: [`../plans/v0.5-slice2-shared-projection.md`](../plans/v0.5-slice2-shared-projection.md)
Parent: [`v0.5-admission-webhook.md`](v0.5-admission-webhook.md) · Context: gap **C2** is
closed by slices **2 → 3 → 4** (shared projection → informer indexed `state.Provider` →
staleness guard); this is slice 2, a no-behaviour-change prerequisite.

## Summary
Factor the per-object relation extraction out of `internal/cluster.BuildObjects` into a
shared, **index-free** `project`, so the informer index (slice 3) and the one-shot reader
produce identical `closure.Object`s — the parity guarantee that keeps goldens 01–27
authoritative for the live paths.

## Interface
```go
// project extracts the four relations from ONE object into a closure.Object using the
// SAME unstructured field paths as the loader's parity oracle. INDEX-FREE: a cross-ref
// carries its referent's Kind/namespace/name with an EMPTY uid. Shared verbatim by the
// one-shot reader (via BuildObjects) and the informer index (slice 3). Fails closed on a
// malformed object (missing kind/name, bad matchExpressions operator).
func project(o unstructured.Unstructured, scope ScopeInfo) (closure.Object, error)

// BuildObjects = map(project) + resolveCrossRefUIDs (the one-shot uid optimization).
func BuildObjects(objs []unstructured.Unstructured, scope ScopeInfo) ([]closure.Object, error)

// resolveCrossRefUIDs fills cross-ref referent uids from the projected set's own real
// uids — the one-shot equivalent of the old nameUIDIndex. Index-free callers (the
// informer index) do not invoke it. Mutates objs in place.
func resolveCrossRefUIDs(objs []closure.Object)
```

## Key decision: the shared projection is INDEX-FREE
The informer sees objects one at a time, so `project` cannot resolve a cross-ref's referent
uid against a whole-set index. It need not: `closure.crossRefMatches(refd, target)` matches
by uid only when **both** carry one, else by Kind/namespace/name. Since Kind/namespace/name
uniquely identifies a live object (and the corpus synthesizes `uid = uid:Kind/ns/name`),
the uid-match and the name-match select the same object → **identical closure result**.
Dropping cross-ref uid resolution from the per-object projection changes the internal
`CrossRef.Ref.UID` field but **no closure/escaping/external set**.

`BuildObjects` keeps the uid fill as a thin post-pass purely as a one-shot optimization, so
existing callers and `TestBuildObjectsCrossRefUIDResolution` are unchanged. Owner edges are
already index-free (`ownersOf` reads `ownerReferences[].uid` off the child).

## Components touched
- `internal/cluster/build.go` — `project`, `resolveCrossRefUIDs`, `BuildObjects` recomposed;
  `newNameUIDIndex`/`buildOne` removed.
- `internal/cluster/crossref.go` — cross-ref extractors lose the `nameUIDIndex` parameter
  and emit empty-uid referents.
- `internal/cluster/projection_test.go` — the project↔BuildObjects parity tests.

## Testability
- `TestProjectMatchesBuildObjectsExceptCrossRefUID` — agree on every relation; project's
  cross-ref uids are empty, and uid-zeroed cross-refs are equal.
- `TestProjectAndBuildObjectsYieldSameClosure` — a State from `project` and one from
  `BuildObjects` give the same `closure.Safe` verdict + closure/escaping/external sets.
- Regression: corpus parity 01–27, cross-ref-uid resolution, robustness, fuzz — all green.

## Alternatives considered
- **Keep BuildObjects index-resolving and give the informer its own separate extractor.**
  Rejected: two extractors drift → the live verdict diverges from goldens. One shared
  index-free `project` with a one-shot post-pass keeps a single source of truth.
- **Resolve cross-ref uids in the informer via the indexes at lookup time.** Unnecessary —
  the Kind/ns/name fallback already gives the correct match; adding uid plumbing buys
  nothing and complicates event handling.
