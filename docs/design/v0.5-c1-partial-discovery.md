# Design: v0.5 Slice 1 — C1 partial-discovery tolerance
Status: draft · Date: 2026-06-26 · Slug: `v0.5-c1-partial-discovery`
Parent design: [`v0.5-admission-webhook.md`](v0.5-admission-webhook.md) (gap **C1**, slice 1)
Plan: [`../plans/v0.5-admission-webhook.md`](../plans/v0.5-admission-webhook.md) (slice 1)

## Problem (C1, Critical)
`internal/cluster/reader.go` calls `disc.ServerGroupsAndResources()` in three places —
`State` (reader.go:99), `ResolveKind` (reader.go:140), and `newDiscoveryScope`
(reader.go:260) — and treats **any** non-nil error as fatal, returning it up the
fail-closed path.

But `ServerGroupsAndResources()` returns a non-nil `*discovery.ErrGroupDiscoveryFailed`
whenever *any* aggregated `APIService` is momentarily unavailable (metrics-server
rolling, a flaky custom-metrics adapter, a cert-manager / Knative / KEDA webhook group
unreachable) **while still returning every group it could resolve**. The current code
discards the resolved lists and fail-closes the whole check.

Real-world impact: on an ordinary production cluster KRSM refuses to evaluate *any*
action the moment an unrelated aggregated API hiccups. "Deny because `metrics.k8s.io`
was down" is indistinguishable from "deny because I couldn't read Pods" — the single
biggest real-world blocker.

## Fix
Detect `*discovery.ErrGroupDiscoveryFailed`, **keep the resolved lists**, and fail-closed
**only if a closure-relevant group is among the failed set**. Any other error type stays
fatal (unchanged fail-closed behaviour).

### The "closure-relevant" predicate
`ErrGroupDiscoveryFailed.Groups` is a `map[schema.GroupVersion]error` naming the groups
that failed. A failed group is **closure-relevant iff its group is a built-in group**
(`builtInGroups`: core `""`, `apps`, `batch`, `policy`, `networking.k8s.io`,
`autoscaling`, `rbac.authorization.k8s.io`, `storage.k8s.io`, `apiextensions.k8s.io`).

Rationale:
- The built-in groups host every kind in `readTargets` (Pod/Service/ConfigMap/Secret/
  PVC/Namespace, Deployment/ReplicaSet/StatefulSet/DaemonSet, Job/CronJob, PDB,
  NetworkPolicy, HPA) — the relations the closure walk fundamentally depends on. They are
  served by the core kube-apiserver, so their discovery failing signals a real apiserver
  problem, not a flaky add-on — failing closed is correct.
- The flaky failures C1 is about (`metrics.k8s.io`, `custom.metrics.k8s.io`,
  `external.metrics.k8s.io`, webhook-backed aggregated groups) are **non-built-in** and
  carry no closure relation, so their failure is tolerated: "an aggregated metrics API
  failing must not deny a `delete deployment`."

### Accepted residual risk (honest bound)
A *non-built-in CRD group* can host a closure participant via `ownerReferences` (the broad
CRD-inclusion rule in `selectTargets`). If such a CRD group's discovery fails
simultaneously, we tolerate it and proceed on the resolved lists — a potential missed
member (a false ALLOW vector). This is the deliberate C1 trade-off: the alternative
(fail-closed on *any* non-built-in discovery hiccup) reproduces the original
unusable-on-a-real-cluster behaviour, because aggregated APIs are exactly the
non-built-in groups that flap. The bound is: built-in relation groups are protected
(fail-closed); add-on/aggregated/CRD-group outages degrade to a tolerated partial read.
Surfaced here rather than hidden. (The informer path in slice 3 reuses the same
predicate at factory startup.)

## Implementation
A single shared helper replaces the three inline `ServerGroupsAndResources()` call sites,
so the tolerance is defined once and cannot drift between the three readers.

```go
// serverResources lists the cluster's API resources with partial-discovery tolerance
// (C1). It keeps the groups discovery COULD resolve and fails closed only when a
// closure-relevant (built-in) group is among the failed set; an aggregated/add-on group
// failing (metrics.k8s.io, a flaky adapter/webhook) is tolerated. Any non-
// ErrGroupDiscoveryFailed error stays fatal.
func serverResources(disc discovery.DiscoveryInterface) ([]*metav1.APIResourceList, error)

// closureRelevantFailures returns the built-in (closure-relevant) GroupVersions from a
// partial-discovery error; a non-empty result means fail closed.
func closureRelevantFailures(gdf *discovery.ErrGroupDiscoveryFailed) []schema.GroupVersion
```

`State`, `ResolveKind`, and `newDiscoveryScope` all switch to `serverResources(disc)`.
The error message names the failed closure-relevant groups (never any credential
material), so an operator can tell a tolerated add-on outage from a real fail-closed deny.

## Testability
Hermetic, via a `partialDiscovery` fake wrapping `*discoveryfake.FakeDiscovery` and
overriding `ServerGroupsAndResources` to return resolved lists **plus** an
`*discovery.ErrGroupDiscoveryFailed` (the real client's partial-failure shape).

- **Tolerated:** an aggregated-group (`metrics.k8s.io`) failure alongside resolved
  core+apps lists → `State` returns a usable closure (an unrelated target still resolves),
  `newDiscoveryScope` returns a populated scope, `ResolveKind` still canonicalises.
- **Fail-closed:** a built-in group (`apps/v1`) among the failed set → `State`,
  `newDiscoveryScope`, and `ResolveKind` all return an error.
- **Unchanged:** a plain (non-`ErrGroupDiscoveryFailed`) discovery error still fails
  closed (existing `erroringDiscovery` tests).
- The error must not leak `token`/`Bearer`.
