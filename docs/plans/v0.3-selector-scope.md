# Plan: v0.3-selector-scope
Status: draft · Date: 2026-06-21 · Slug: `v0.3-selector-scope`

## Problem
v0.2 (ADR-0005) made the **closure** side bind label selectors precisely — `matchLabels`
*and* `matchExpressions`, with absence-sensitive operators. But the **scope** side is still
flat identity: `ScopeRef{GVK, Namespace, Name-glob}` (`closure/model.go:111`), matched by
`matchScope` which sees only a `Ref` and compares name with `path.Match` (`closure/safe.go:62`).

So scope cannot express "the Pods matching `app=web`", even though the closure already binds
exactly that set. Testing a precise closure against an imprecise scope produces **avoidable
false Blocks**: a task whose real intent is selector-shaped must be written as a brittle
name-glob, and any closure member it can't enumerate escapes → wrong Block. Selector precision
on only one side of `C ⊆ scope(T)` is half a fix. ADR-0005 names closing this asymmetry as the
next correctness-relevant step, and ROADMAP v0.3 makes it the slice that *leads* the milestone.

## Goals / Non-goals
- **Goal:** add a **selector** scope dimension carrying a `closure.LabelSelector`
  (`matchLabels` + `matchExpressions`), reusing the existing selector type (one implementation,
  ADR-0005).
- **Goal:** make `matchScope` **state-dependent** — a selector clause matches a closure member
  by reading *that member's* labels from `State`.
- **Goal:** the loader (`internal/scenario.parseScope`) parses the selector clause from
  `scope.yaml`.
- **Goal:** a new golden scenario that is **wrongly Blocked today** purely because scope can't
  express the selector the closure binds, and classifies correctly once the dimension exists.
- **Non-goal:** the full `TaskContract` → `ScopePredicate` compiler and the `scope/` package
  (the other v0.3 bullets — later slices).
- **Non-goal:** the other state-dependent dimensions (`ownership`, `reference`) — selector only.
- **Non-goal:** inexpressible-scope **gap signalling** (e.g. set-difference "all `app=web`
  except canaries"). A positive selector clause only; set-difference is a later v0.3 bullet.
- **Non-goal:** any CRD/YAML wire schema for real `TaskContract`s; this stays at the corpus
  `scope.yaml` level.

## Current state
- `closure/model.go:109-115` — `ScopeRef{GVK, Namespace, Name}`, flat identity, "v0.1".
- `closure/safe.go:17` — `Safe(s State, a Action, scope []ScopeRef)` already holds `State`.
- `closure/safe.go:62-81` — `matchScope(r Ref, scope []ScopeRef) bool`: GVK + namespace gating
  then `path.Match(sc.Name, r.Name)`. No label/selector awareness, no `State`.
- `closure/selector.go` — `LabelSelector{MatchLabels, MatchExpressions}` with a stdlib
  `Matches(labels map[string]string) bool`. **Reuse verbatim.**
- `closure/state.go:6-32` — `State.Get(Ref) (*Object, bool)`; `Object.Labels` (`model.go:80`)
  carries the resource's labels. This is the seam: a closure member is always resolvable in
  `State` (the closure is walked *from* `State`).
- `internal/scenario/scenario.go:450-462` — `parseScope` unmarshals `scope:` as `[]rawRef`
  ({group,version,kind,namespace,name}). No selector field today.
- Corpus: `closure/testdata/scenarios/<NN-name>/{cluster,request,scope,expected}.yaml`
  (19 scenarios), asserted by the `internal/scenario` golden tests. `scope.yaml` is
  `scope: [ {kind, namespace, name}, ... ]`; `expected.yaml` carries `verdict/closure/escaping/
  external`.

## Approach
Extend the scope clause from *identity-only* to *identity-or-selector*, and give `matchScope`
read access to `State` so a selector clause can be evaluated against a candidate's live labels.

- **One selector implementation.** A selector scope clause holds a `closure.LabelSelector`;
  matching reuses `LabelSelector.Matches`. No second selector engine, no `k8s.io/...` — the
  `closure` SDK stays stdlib-only (ADR-0002/0005, enforced by `internal/archguard`).
- **State-dependent matching, minimal seam.** `matchScope` gains access to label lookup. A
  selector clause matches `r` when GVK/namespace gate as today **and** `r`'s labels (from
  `State.Get(r)`) satisfy the clause's selector. Identity clauses are unchanged → the 19
  existing scenarios are untouched.
- **Fail-closed on unresolvable labels.** If a selector clause's candidate cannot be resolved in
  `State`, the clause does **not** match (no over-grant) — consistent with DESIGN §5.
- **Loader mirrors the engine.** `parseScope` learns the selector clause shape so the new
  scenario (and future `TaskContract`s) can declare it.

**Clause shape follows DESIGN §6, not a flat field.** DESIGN §6 already commits scope to
"dimension-typed allow-clauses, one per relation dimension plus a flat identity dimension"
(`dim: resource`, `dim: selector`, …), and ROADMAP v0.3 enumerates the five dimensions. So this
slice introduces the **discriminated clause** with the two dimensions it needs now (`resource`
= today's identity clause; `selector` = the new one); the later compiler slice *extends* it with
`ownership`/`namespace`/`reference` rather than reworking a flat shape. This is barely larger
than an optional field and keeps the first slice on the design's committed trajectory.

One seam decision is deferred to the **design-doc** (see Open questions): how `matchScope`
reaches labels (pass `State`, pass a narrow `func(Ref)(map[string]string,bool)` lookup, or
pre-resolve in `Safe`).

## Steps (vertical slices)
Each slice is independently shippable and demoable.

1. [ ] **Selector matching in the engine.** Add the selector dimension to the scope clause and
   make `matchScope` state-aware; identity clauses unchanged. Table-driven `closure`-package
   unit tests covering: `matchLabels` hit/miss, `matchExpressions` (`In`/`NotIn`/`Exists`/
   `DoesNotExist`, incl. absence), GVK/namespace gating still applies, unresolvable candidate →
   no match, and an empty/nil selector clause behaves per the design decision. **Demo:**
   `go test ./closure -run Scope -race`.
2. [ ] **Loader + golden scenario.** Teach `parseScope` the selector clause shape. Add the new
   corpus scenario (`closure/testdata/scenarios/<NN>-scope-selector-precision/`) that is wrongly
   Blocked today and now classifies correctly; assert via the `internal/scenario` golden tests;
   confirm all 19 existing scenarios stay green. **Demo:** `krsm check
   closure/testdata/scenarios/<NN>-scope-selector-precision` flips Block → Allow.

## Risks & unknowns
- **Public SDK shape change.** `ScopeRef` (and possibly `matchScope`'s signature, though it is
  unexported) change shape — a pre-1.0 breaking change to the `closure` API. Acceptable now with
  no external importers (same rationale as ADR-0005's `Object.Selector` retype); record it in
  the design and CHANGELOG.
- **Over-grant via empty selector.** A present-but-empty selector clause must **not** silently
  mean "match everything" (unlike `selectorBinds`' kind-aware binding rule). Decide explicitly:
  match-nothing or load-time rejection. Wrong choice = a silent scope blow-out.
- **Regression surface.** The 19 goldens must stay byte-for-byte; the change must be additive on
  the identity path.
- **Scenario realism.** The new scenario must be genuinely "wrongly Blocked *today* purely
  because scope can't express the selector" — not blocked for some other reason — or it doesn't
  prove the fix. Verify by running it pre-change (expect Block) and post-change (expect Allow).

## Definition of done
- A `scope.yaml` selector clause compiles and correctly classifies the corpus, **including** a
  new scenario wrongly Blocked today purely because scope cannot express the selector the closure
  already binds.
- All 19 pre-existing scenarios remain green.
- `closure` stays stdlib-only (`internal/archguard` passes).
- `go test -race ./...`, `golangci-lint`, and `staticcheck` are clean (`make check`).
- The change is recorded in `CHANGELOG.md`; if it constitutes a model decision, an ADR follows
  (selector scope dimension as ADR-0005 follow-on).

## Open questions
1. **`matchScope` ↔ `State` seam:** pass `State`, a narrow label-lookup func, or pre-resolve
   labels in `Safe`? (design-doc decides; leaning narrow label-lookup to keep `matchScope`
   testable without a full `State`.)
2. **Clause encoding:** RESOLVED in Approach — use the `dim`-discriminated clause
   (`resource` | `selector`) per DESIGN §6, not a flat optional field. Remaining sub-question for
   design-doc: keep `ScopeRef` as the `resource`-dim representation and add a sibling
   `selector`-dim type, or introduce one `ScopeClause` sum type now? (Leaning: one `ScopeClause`
   with a `Dim` tag, so `[]ScopeRef` → `[]ScopeClause` once, cleanly.)
3. **Empty/nil selector clause:** match-nothing vs reject-at-load. (Recommend match-nothing in
   the engine + a loader warning, to stay fail-safe.)
4. **Selector clause beyond Pods:** DESIGN §6's example gvk-gates the selector clause to `Pod`.
   Do we allow a selector clause on any kind that carries labels, or restrict to Pods for this
   slice? (Recommend: gate by the clause's `gvk` like identity clauses; no special Pod-only
   rule — the selector simply tests `r.Labels`.)
