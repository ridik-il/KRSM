# Plan: v0.5-admission-webhook
Status: draft · Date: 2026-06-24 · Slug: `v0.5-admission-webhook`

## Problem
v0.4 shipped the offline closure engine, derived default scope, and a **one-shot live
reader** that makes `krsm check` work against a `kind` cluster (ROADMAP v0.4 "Done
when" — proven by `internal/e2e`). But v0.4 is the *kind milestone*, not a production
control. A critical real-world review of the v0.4 live path (PR #11) found that the
one-shot `internal/cluster.Reader` will **misbehave on an ordinary production cluster**:

- **C1 — partial discovery is fatal.** `disc.ServerGroupsAndResources()` returns a
  non-nil `*discovery.ErrGroupDiscoveryFailed` whenever *any* aggregated APIService is
  momentarily down (metrics-server rolling, a flaky custom-metrics adapter, a webhook
  group unreachable). `internal/cluster/reader.go:99-102` treats *any* error as fatal and
  fail-closes the whole check — so KRSM refuses to evaluate *any* action on a normal
  cluster the moment an unrelated aggregated API hiccups.
- **C2 — no consistent snapshot.** Each GVR is a separate `List` at its own
  resourceVersion (`reader.go:111-118`); the object set is not point-in-time, so a
  neighbour created/deleted mid-read yields a closure over torn state — a **false ALLOW
  (missed collateral)**, the one error class a safety gate must not have.
- **C3 — unbounded whole-cluster list, including every Secret's `data`.**
  `List(ctx, metav1.ListOptions{})` has no pagination and no field scoping (`reader.go:113`);
  one check pulls every Pod and **every Secret's full `data`** into memory — a latency/OOM
  hazard and a real secret-exposure surface (the closure only needs Secret *metadata*).
- **S1 — no timeout/cancellation.** `cmd/krsm/main.go:260` uses `context.Background()`;
  a hung API server hangs `krsm check` forever.
- **S2 — group-blind target resolution.** `ResolveKind` discards the API group and the
  human key is group-agnostic, so same-Kind-across-groups collides in CRD-heavy clusters.
- **S3 — multi-version resources are double-listed.** `ServerGroupsAndResources` returns
  every served version; `selectTargets` dedups only by GVR, so a resource at two versions
  (e.g. `HorizontalPodAutoscaler` at `autoscaling/v1` *and* `/v2` — a built-in in the
  default set) is listed twice. The uid-keyed closure walk dedups the *verdict*, so this
  is waste + a footgun, not a wrong answer — fix by listing preferred versions only.
- **S4 — cascade is hardcoded.** The CLI always sets `Cascade: true` for delete; a real
  delete's `propagationPolicy` (Background/Foreground/**Orphan**) is not expressible.

v0.5 is the **admission webhook** (ROADMAP v0.5, DESIGN §5, ADR-0011 audit-first) — the
actual product: flag/deny escaping actions *before they persist*. Crucially, the webhook
**cannot be built on top of the one-shot reader**: DESIGN §4 ("too slow for an admission
webhook at 10⁵ resources") and ADR-0004 (informer-backed indexed state, fail-closed)
already decided that the hot path is **informer-fed inverted indexes**, costing `O(c·d)`
(blast radius), not `O(c·n)` (cluster size). That informer state path is *also the
natural, correct fix* for C2 (a single consistent watch-fed cache), C3 (metadata-only
informers for Secrets; list-watch paginates internally), S3 (one informer per preferred
version), and — because an `AdmissionReview` carries the real GVK + uid + `DeleteOptions`
+ old/new payload — S2 and S4 disappear on the webhook path. C1 and S1 need explicit
handling regardless. So v0.5 = **fix the v0.4 real-world read gaps, build the informer
state path, and put the audit-first webhook on top of it.**

## Goals / Non-goals
- **Goal (v0.4 hardening, prerequisite):** make the *existing* one-shot CLI read path
  honest on a real cluster — tolerate partial discovery (C1), add a deadline + signal
  cancel (S1), list preferred versions only (S3), paginate (C3-partial). These are cheap,
  high-value, and de-risk reusing the GVR-selection logic in the informer path.
- **Goal:** a new **`state/` package** — an informer-backed, incrementally **indexed**
  `closure.State` implementation (ADR-0004, DESIGN §4/§7): owner index, namespace index,
  selector inverted-index, reverse cross-ref index; `WaitForCacheSync` before serving;
  **metadata-only** informer for Secrets so `data` is never cached (C3); one informer per
  **preferred** version (S3); bounded resync. A *second* `closure.State` beside
  `NewScanState`/`internal/cluster`, behind the unchanged interface seam.
- **Goal:** a **per-request staleness guard** (C2, ADR-0004 open question): compute
  closure against the index, detect drift against the admission request's own object
  (`resourceVersion`/`generation`), and **fail-closed on unresolved drift** with a bounded
  on-demand `GET` fallback over the *closure neighbourhood* only (never the whole cluster).
- **Goal:** a **`webhook/` package** — a `ValidatingWebhook` HTTPS server that decodes an
  `AdmissionReview`, extracts the `closure.Action` from it (real GVK + uid; **real
  `propagationPolicy` from `DeleteOptions`** → fixes S4; **old/new from the request** →
  enables the selector/finalizer-mutation verdicts the CLI could not express), runs the
  **shared** `closure.Safe` → `scope.Mode.Apply`, and answers allow/deny with a reason +
  scope provenance. **Fail-closed** on any internal error, cache-not-synced, or handler
  timeout (S1). Strict `AdmissionReview` schema validation (DESIGN §9).
- **Goal:** **request scoping** — `namespaceSelector`/`objectSelector` + an agent-identity
  matcher so KRSM gates only agent-originated requests, not humans or cluster controllers.
- **Goal:** **progressive scope channel** (ADR-0011) on the webhook: Level 0 derived
  (reuse `scope.Derive`), **Level 1** `krsm.io/target` annotation (root the derivation at
  the declared task target), **Level 3** `TaskContract` CRD resolved from an informer
  (ADR-0003/0009 `scope.Compile`); provenance (`derived` | `annotation` | `template` |
  `contract`) reported in the response. Level 2 templates land in v0.6 (separate).
- **Goal:** **audit-first** (ADR-0011): install default `audit` (a derived-scope escape is
  a `Warn`/allow-with-warning, never a deny); `enforce` (deny pre-persistence) opt-in.
  Reuse `scope.Mode`. A **cross-boundary allowlist** (namespaces + cluster-scoped kinds)
  exempts legitimate shared-resource references from *derived* (Level 0/1) modes.
- **Goal:** **deployable** — `ValidatingWebhookConfiguration`, TLS/cert management
  (cert-manager-friendly + a self-bootstrapped CA path), least-privilege RBAC for the
  informer set + the `TaskContract` CRD, and a Helm chart whose default is audit +
  `failurePolicy: Fail` scoped tightly enough not to gate `kube-system`/controllers.
- **Goal:** the ROADMAP v0.5 "Done when" proven on `kind`, end-to-end, both modes, both
  derived and full-contract.
- **Non-goal:** Level 2 templates (ADR-0011 Level 2 → **v0.6**, pure compiler sugar).
- **Non-goal:** the `reference` scope dimension (deferred since v0.3).
- **Non-goal:** mutating admission — KRSM *decides*, it does not rewrite (DESIGN §1/§5).
- **Non-goal:** any change to `closure.Safe`'s signature, the closure walk, the
  derived-scope/mode code, or goldens 01–27. The webhook *reuses* them.
- **Non-goal:** HA/leader-election, multi-replica cache coherence tuning, and the
  10²–10⁵ scale study (research apparatus; a single-replica audit-first webhook is the cut).
- **Non-goal:** the scope-channel *trust* model (who may sign a `TaskContract`) beyond a
  documented threat-model stub — ADR-0003 defers it to SECURITY.md; v0.5 reports
  provenance but does not yet verify issuer identity.

## Current state
- `closure/state.go` — the `State` interface (`Get`, `OwnedChildren`, `NamespaceContents`,
  `PodsSelectedBy`, `PodsMatching`, `SelectorsTargeting`, `SelectorsMatchingLabels`,
  `Consumers`, `ControllersTargeting`) and the linear-scan `scanState`/`NewScanState`. The
  v0.1 doc comment already anticipates "a later informer-backed indexed implementation
  satisfies the same interface without changing callers" — that implementation is this
  slice. The informer state must satisfy this **same interface**.
- `closure/safe.go` / `closure/closure.go` — `Safe(State, Action, []ScopeClause)`,
  fail-closed on empty closure; the BFS uses only `State` methods and dedups by
  `Ref.key()` (uid), so a faithful indexed `State` plugs in unchanged.
- `closure/model.go` — `Action{Verb, Target, Cascade, Old, New}`. `Old/New` carry the
  mutation payload the CLI could not supply but the `AdmissionReview` can.
- `scope/scope.go` — `Derive(target) ScopePredicate` (Level 0), `Mode{Audit,Enforce}` +
  `Apply`, `Compile` (Level 3). All reused; provenance constants exist.
- `internal/cluster/{reader,build,index}.go` — the one-shot live reader + the pure
  `BuildObjects` projection + discovery-backed `ScopeInfo`. The relation-extraction logic
  (`BuildObjects` and its unstructured field paths) is the **reference the informer index
  must reproduce**; the GVR-selection + scope logic (`selectTargets`/`scopeFromLists`/
  `supportsList`) is reused after the C1/S3 hardening.
- `internal/scenario` — the YAML loader (golden tests + CLI dir mode). Unchanged.
- `internal/e2e/cluster_e2e_test.go` — the v0.4 `//go:build cluster` kind test; v0.5 adds
  a sibling webhook e2e.
- `internal/archguard/archguard_test.go` — `closure`/`scope` must stay stdlib-only. The
  new `state/` and `webhook/` packages sit **outside** both trees (DESIGN §7 names `state/`
  and `webhook/` as non-embeddable). client-go already in `go.mod` (v0.4).
- `deploy/rbac/krsm-reader-clusterrole.yaml` — the v0.4 read-only ClusterRole; v0.5
  extends it (TaskContract get/list/watch) and adds the webhook deployment manifests.
- `go.mod` — `k8s.io/client-go` v0.34.9, `k8s.io/apimachinery` v0.34.9 (go 1.25.0). v0.5
  adds `k8s.io/api` (typed `admission/v1`, `meta/v1` metadata informers) and the
  controller-runtime *or* raw client-go informer surface (decision below); CI's
  `govulncheck`/`dependency-review`/CodeQL/Scorecard must stay green.

## Approach
Build bottom-up so each slice is observable and the webhook lands last on a proven state
path. The spine: **harden the read → build the indexed informer `State` → guard its
freshness → serve it through the webhook → wire the scope channel → ship the deployment →
prove on kind.**

### Decisions taken
1. **Fix the v0.4 read gaps first, in `internal/cluster`, before the webhook (C1/S1/S3).**
   They are cheap, make the CLI itself real-cluster-honest, and the hardened GVR-selection
   is reused by the informer factory. C1: detect `discovery.ErrGroupDiscoveryFailed`,
   keep the resolved groups, and fail-closed **only** if a closure-relevant group is among
   the failed set (an aggregated metrics API failing must not deny a `delete deployment`).
   S3: `ServerPreferredResources()` (preferred version per resource) instead of
   `ServerGroupsAndResources()` for *listing* (scope discovery still enumerates all for the
   namespaced flag). S1: a default `--timeout` (e.g. 30s) + `signal.NotifyContext`.
2. **Informer-backed indexed `State` is the hot path (ADR-0004, DESIGN §4).** Not a
   one-shot list. The four inverted indexes are maintained by informer event handlers
   (add/update/delete) keyed for `O(c·d)` closure. The pure `BuildObjects` relation
   extraction is **factored into a shared per-object projection** so the informer index and
   the one-shot reader produce identical `closure.Object`s from the same unstructured input
   (the parity guarantee that keeps goldens authoritative).
3. **Secrets via a metadata-only informer (C3).** Use a `metadatainformer.SharedInformerFactory`
   (`PartialObjectMetadata`) for `Secret` (and any kind whose only role in the four
   relations is being *referenced*, not selected-by-labels), so Secret `data` is never
   fetched or cached. Kinds whose labels matter for selector binding (Pods) need full
   objects; the factory is split accordingly. This both removes the exposure surface and
   shrinks cache memory.
4. **Per-request staleness guard, bounded fallback (C2).** Primary read is the index.
   The admission request carries the *authoritative* current target object; compare its
   `resourceVersion` against the cached copy. On a detected gap (request newer than cache),
   do a **bounded live `GET`** over the closure neighbourhood (the `O(d)` members), not a
   re-list; if still unresolved, **fail-closed** (ADR-0004). Because the webhook only fires
   on agent requests (low QPS), the occasional bounded fallback is affordable.
5. **Webhook framework: raw client-go `admission/v1` + `net/http`, not controller-runtime.**
   Lean: KRSM needs an HTTPS handler, informers (client-go `SharedInformerFactory`), and
   `AdmissionReview` decode — all in client-go/api already in `go.mod`. controller-runtime
   would pull a large manager/reconciler surface KRSM doesn't use and enlarge the
   `govulncheck` tree. (Revisit if CRD/cert plumbing proves heavier than expected.)
6. **TLS/cert: cert-manager-friendly + a self-bootstrap fallback.** The chart supports a
   `cert-manager` `Certificate` (preferred) and a self-signed CA bootstrap job that patches
   the `caBundle` (for clusters without cert-manager). The webhook never serves plaintext.
7. **`failurePolicy: Fail` (fail-closed) with a tight `namespaceSelector`/`objectSelector`.**
   Fail-closed is the whole point (DESIGN §5), but a fail-closed webhook that matches too
   broadly can wedge the cluster (the classic admission-webhook outage). The match is
   restricted to agent-originated requests (an identity label/annotation + a namespace
   selector that **excludes** `kube-system` and KRSM's own namespace), and the install docs
   call out the blast radius explicitly.
8. **Mode + cross-boundary allowlist are config, not new engine.** Reuse `scope.Mode`;
   the allowlist is a startup config (namespaces + cluster-scoped kinds) consulted only in
   *derived* (Level 0/1) modes before counting a member as escaping. A full contract
   (Level 3) states its own scope and ignores the allowlist.

### Dependency / supply-chain position
- Add `k8s.io/api` (typed `admission/v1` for the review; `meta/v1` `PartialObjectMetadata`
  for metadata informers) — already an indirect dep, promote to direct. Add the client-go
  `metadatainformer`/`dynamicinformer` + `tools/cache` surface (already in the client-go
  tree). No controller-runtime (decision 5).
- These live **only** under `state/`, `webhook/`, `cmd/krsm` — `archguard` keeps them out
  of `closure`/`scope`. Confirm `go list -deps ./closure/... ./scope/...` stays k8s-free.
- `govulncheck` over a slightly larger tree: resolve any advisory by **bumping**, never
  suppressing (v0.4 precedent). `dependency-review` denies GPL/AGPL — client-go/api are
  Apache-2.0. Note the new admission attack surface in the PR (DESIGN §9).

## Steps (vertical slices)
Each slice ends in something observable; the early ones are hermetic (fake clients), the
cluster-touching behaviour is proven last behind `//go:build cluster`. Each goes through
`design-check-tdd → tdd`; `make check` green after each.

1. [ ] **v0.4 read hardening (C1 + S1 + S3 + C3-pagination) in `internal/cluster` + CLI.**
   Tolerate `discovery.ErrGroupDiscoveryFailed` (proceed on resolved groups; fail-closed
   only if a closure-relevant group failed); switch listing to `ServerPreferredResources`;
   add `--timeout` + `signal.NotifyContext` to the live path; paginate `List` with `Limit`/
   continue. Hermetic fake-discovery/dynamic tests for each. *Demo: a fake discovery set
   with one failed aggregated group still produces a verdict for an unrelated target; a
   multi-version resource is listed once.*
2. [ ] **Shared per-object projection.** Factor the `BuildObjects` relation extraction into
   a per-object `project(unstructured, ScopeInfo) (closure.Object, error)` reused by both
   the one-shot reader and the informer index, with a parity test that both paths yield the
   same `closure.Object` for the same input. *Demo: table test — reader and (stub) index
   agree object-for-object on the scenario-01 fixtures.*
3. [ ] **Indexed informer `state.Provider` implementing `closure.State`.** New `state/`
   package: a `SharedInformerFactory` (+ `metadatainformer` for Secrets, decision 3) over
   the preferred-version tracked GVRs; four inverted indexes maintained by event handlers;
   `WaitForCacheSync`; serve the nine `closure.State` methods from the indexes in `O(d)`.
   Unit-test each index and each `State` method with a fake clientset/informer; assert the
   indexed `State` produces the **same closure/escaping set as `NewScanState`** for the
   golden corpus (the parity oracle, reused). *Demo: feed the corpus objects through the
   informer index, get goldens 01–27 verdicts; Secret objects carry no `data` in cache.*
4. [ ] **Staleness guard + bounded neighbourhood fallback (C2).** Add the
   resourceVersion-drift check and the `O(d)` on-demand `GET` fallback; fail-closed on
   unresolved drift, with a distinct reason. Table tests with a fake client returning a
   newer object than the cache. *Demo: a stale-cache scenario triggers a bounded refresh,
   not a whole-cluster re-list; an unresolvable drift fails closed.*
5. [ ] **`webhook/` server (audit-first, fail-closed).** HTTPS `net/http` server;
   `admission/v1` `AdmissionReview` decode with strict schema validation; extract `Action`
   (real GVK+uid; `propagationPolicy` from `DeleteOptions` → S4; `Old/New` from
   `object`/`oldObject` → mutation verdicts); request scoping via `namespaceSelector`/
   `objectSelector` + agent-identity matcher; run shared `Safe` → `Mode.Apply`; respond
   allow/deny with reason + provenance; per-request deadline + fail-closed on
   error/not-synced/timeout (S1). Hermetic: drive `Handle` with crafted `AdmissionReview`
   JSON over a fake state. *Demo: a synthetic escaping `delete` AdmissionReview returns
   `allowed:false` under enforce and `allowed:true`+warning under audit.*
6. [ ] **Progressive scope channel on the webhook (Level 0/1/3 + provenance + allowlist).**
   Level 0 `scope.Derive`; Level 1 `krsm.io/target` annotation roots the derivation; Level
   3 `TaskContract` CRD via an informer → `scope.Compile`; cross-boundary allowlist for
   derived modes; provenance in `Decision.Reason`/the response. *Demo: the same escaping
   action is Allowed under a full contract that authorises the tree, Warned under derived
   audit, and Denied under derived enforce — provenance shown in each.*
7. [ ] **Deployment: cert, RBAC, `ValidatingWebhookConfiguration`, Helm.** TLS via
   cert-manager `Certificate` + a self-bootstrap CA fallback; least-privilege RBAC (the
   informer set get/list/watch + TaskContract); the webhook config with a tight
   `namespaceSelector`/`objectSelector` and `failurePolicy: Fail`; a Helm chart defaulting
   to audit. `helm template` + `kubeconform`/`kubeval` in CI. *Demo: `helm template` renders
   a valid, audit-default, tightly-scoped config; RBAC has no write/wildcard verb.*
8. [ ] **`kind` acceptance e2e (proves ROADMAP v0.5 "Done when").** `//go:build cluster` +
   `make e2e` extension: stand up kind, install the webhook (audit), apply a
   scenario-01-equivalent fixture + an agent identity, issue an agent-stamped escaping
   `delete` and an in-scope one; assert **audit → flagged-but-admitted**, **enforce →
   denied pre-persistence**, in-scope → admitted, for **both** a derived scope (no contract)
   and a full `TaskContract`. *Demo: `make e2e` green; the API server rejects the escaping
   delete in enforce mode with KRSM's reason.*

## Risks & unknowns
- **Cache staleness vs the watch-write race (highest, ADR-0004).** The informer cache is
  eventually consistent; a neighbour can change between the snapshot and persistence. The
  staleness guard (slice 4) bounds it but cannot eliminate it without a global snapshot
  primitive Kubernetes lacks. Mitigation: bounded neighbourhood `GET` + fail-closed on
  detected drift; document the residual window; the webhook's low QPS makes the fallback
  affordable. This is the research-relevant SRQ on underdetermination, surfaced honestly.
- **Index/loader parity (high).** The informer index must reproduce `BuildObjects`'s four
  relations exactly or the live verdict diverges from the goldens. Mitigation: slice 2's
  shared projection + slice 3's corpus parity oracle, before any cluster is touched.
- **A fail-closed webhook can wedge the cluster (high, operational).** `failurePolicy: Fail`
  + a too-broad match = an outage if KRSM is down. Mitigation: tight agent-only
  `namespaceSelector`/`objectSelector` excluding `kube-system`/KRSM's own namespace; a
  documented break-glass (delete the `ValidatingWebhookConfiguration`); a liveness/readiness
  probe gated on cache-sync; single-replica is acceptable for audit-first.
- **Cert lifecycle.** A rotated/expired serving cert silently breaks admission. Mitigation:
  cert-manager preferred (auto-rotation); the self-bootstrap path documents manual rotation;
  a startup check that the `caBundle` matches the serving cert.
- **Metadata-vs-full informer split correctness (C3).** If a kind needs labels for selector
  binding but is fetched metadata-only, selector binding silently breaks. Mitigation:
  `PartialObjectMetadata` *does* carry `labels` — so metadata informers are safe for
  selector-relevant kinds too; only `Secret.data`/`ConfigMap.data` are dropped, and those
  are never read by the four relations. Pin this with a test that a metadata-only Pod still
  binds a Service selector.
- **govulncheck noise from `k8s.io/api`/admission.** Resolve by bump, never suppress.
- **Discovery breadth for informers.** Starting an informer per tracked GVR on a cluster
  missing a group must not crash startup; reuse the C1 partial-discovery tolerance.

## Definition of done
Tied to ROADMAP v0.5 "Done when":
- On `kind`, an agent-stamped **escaping `delete` is flagged in audit mode and denied
  pre-persistence in enforce mode**, and an in-scope one is **admitted** — both with a
  *derived* scope (no contract) and with a full `TaskContract`, end-to-end (slice 8).
- The webhook **fails closed** on cache-not-synced, internal error, handler timeout, and
  detected staleness drift — each with a distinct, credential-free reason (DESIGN §5/§9).
- The **v0.4 real-world read gaps are resolved**: partial discovery tolerated (C1), no
  unbounded/secret-data reads on the hot path (C3), consistent-enough reads with a
  staleness guard (C2), a deadline (S1), preferred-version listing (S3); S2/S4 are moot on
  the webhook path (request carries GVK/uid/`DeleteOptions`/payload) and S2 documented for
  the CLI.
- `closure` and `scope` remain **stdlib-only** (`internal/archguard` green); `state`/
  `webhook` live outside both trees.
- `closure.Safe`'s signature, the closure walk, the derived-scope/mode code, and goldens
  01–27 are **unchanged**; the indexed `State` passes the corpus parity oracle.
- `make check` green (`gofmt`, `go vet`, `golangci-lint`, `staticcheck`, `go test -race`);
  CI green including `govulncheck`, `dependency-review`, CodeQL, Scorecard; the e2e gated
  behind `make e2e` (no kind in CI yet — a job stub is left, per v0.4 precedent).

## Open questions
1. controller-runtime vs raw client-go (decision 5 leans raw) — revisit if cert/CRD
   plumbing is heavier than expected.
2. cert-manager hard dependency vs self-bootstrap default (decision 6 ships both) — which
   is the documented default for the quickstart?
3. Staleness fallback aggressiveness (slice 4): always-GET-the-target vs only-on-detected-
   drift. Lean drift-triggered to keep the hot path read-free; measure if false fail-closeds
   appear.
4. TaskContract CRD versioning: ship `krsm.io/v1alpha1` now and reserve `v1` (DESIGN §6
   shows `krsm.io/v1`)? Lean `v1alpha1` until the schema settles in v0.6 templates.
5. Single-replica only for v0.5, or leader-elected from the start? Lean single-replica
   (audit-first, low QPS); multi-replica cache coherence is a v0.6+/HA concern.
6. Does the cross-boundary allowlist belong in the `TaskContract`/CRD or in webhook startup
   config? Lean startup config for derived modes (it is operator policy, not task scope).
