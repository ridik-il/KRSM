## Problem
The informer-backed `state.Provider` (slice 3, the consistent watch-fed cache that closes

## Approach
`crossRefMatches` (closure/state.go) matches a cross-ref to its referent by uid **only
when both carry one**, else by Kind/namespace/name. Resolving a cross-ref's referent uid
is therefore an *optimization*, not a correctness requirement: Kind/namespace/name
uniquely identifies a live object, so the uid-match and the name-match select the same
object and yield the same closure. This makes an index-free projection sound:

1. `project` extracts Ref/Labels/Selector/Owners/CrossRefs/Finalizers from one object;
   cross-refs carry the referent's Kind/ns/name with an **empty** uid.
2. `BuildObjects` = `map(project)` then a thin `resolveCrossRefUIDs` post-pass that fills
   cross-ref uids from the projected set's own real uids â€” reproducing today's behaviour
   exactly (so `TestBuildObjectsCrossRefUIDResolution` is unchanged). The informer index
   skips the post-pass.
3. Owner edges are already index-free (`ownersOf` reads `ownerReferences[].uid` off the
   child); unaffected.
