# Plan: v0.3-taskcontract-compiler
Status: approved · Date: 2026-06-21 · Slug: `v0.3-taskcontract-compiler`

## Problem
v0.3's theme is "make scope declarable beyond flat lists." Slice 1 (selector-scope) gave the
*engine* the dimension-typed `closure.ScopeClause` and the `resource`/`selector` dimensions, but
there is still **no user-facing scope contract**: scope is assembled ad hoc by
`internal/scenario.parseScope` into a `[]closure.ScopeClause`. ADR-0003 decided scope is supplied
via a **`TaskContract`** the agent references; DESIGN §6 specifies its CRD-shaped YAML; DESIGN §7
puts a **`scope/` package** ("TaskContract → ScopePredicate compiler", *public, embeddable*) in the
layout. Neither exists yet. This slice builds that contract and compiler for the two dimensions we
already have, establishing the package boundary and the compile seam the remaining dimensions plug
into.

## Goals / Non-goals
- **Goal:** a new public **`scope/`** package with a Go `TaskContract` struct (DESIGN §6 shape) and
  `func Compile(TaskContract) (ScopePredicate, error)`.
- **Goal:** `Compile` supports the existing `resource` + `selector` dimensions, building clauses via
  `closure.ResourceClause`/`SelectorClause` and `ScopeClause.Validate()`; an unknown/invalid dim is a
  **compile error** (fail-closed, consistent with slice-1 F1).
- **Goal:** `scope/` stays **stdlib-only & embeddable** (DESIGN §7) — `Compile` takes the *struct*,
  not YAML bytes; YAML→`TaskContract` parsing lives in the loader. Extend `archguard` to guard it.
- **Goal:** end-to-end — a **`TaskContract` YAML** loads, compiles, and classifies the corpus,
  *including* the selector scenario (20). (This satisfies ROADMAP v0.3's milestone done-when.)
- **Non-goal:** the `ownership`, `namespace`, `reference` dimensions — later slices (shape must leave
  room for them).
- **Non-goal:** inexpressible-scope **gap signalling** (DESIGN §6) — a later slice.
- **Non-goal:** a real Kubernetes CRD / apiserver types / the informer scope channel (v0.4–0.5);
  this stays a Go struct + corpus YAML.
- **Non-goal:** changing `closure.Safe`'s signature again — it keeps taking `[]closure.ScopeClause`.
- **Non-goal:** removing the existing `scope.yaml` corpus path — back-compat stays.

## Current state
- **No `scope/` package** (verified).
- `closure/model.go` — `ScopeClause{Dim,GVK,Namespace,Name,Selector}`, `DimResource`/`DimSelector`,
  `ResourceClause()`, `SelectorClause()`, `ScopeClause.Validate()`.
- `closure/safe.go:17` — `Safe(s State, a Action, scope []closure.ScopeClause) Decision`.
- `internal/scenario/scenario.go:464` — `parseScope([]byte) ([]closure.ScopeClause, error)` from
  `scope: [...]` YAML; `Scenario{State,Action,Scope}` (`Load` reads cluster/request/scope.yaml).
- `internal/archguard/archguard_test.go` — runs `go list -deps closure/...` and fails on any dep
  whose first path segment has a dot (a domain). Naively extending to `scope/` would flag the
  legitimate intra-module `github.com/ridik-il/krsm/closure` import — the check must **allowlist the
  module prefix** `github.com/ridik-il/krsm/` while still forbidding external (`k8s.io/...`) deps.
- DESIGN §6 TaskContract: `apiVersion: krsm.io/v1`, `kind: TaskContract`, `metadata{name,namespace}`,
  `spec.allow: [ {dim:…} ]`, `maxSeverity`.

## Approach
Add a thin compiler in front of the existing engine; do not re-model scope.

- **`scope.TaskContract`** mirrors DESIGN §6: `APIVersion`, `Kind`, `Metadata{Name,Namespace}`,
  `Spec{Allow []AllowClause, MaxSeverity}`. `AllowClause` is dim-typed (resource/selector now;
  fields for ownership/etc. added by later slices). Stdlib types only.
- **`scope.Compile(tc)`** validates `apiVersion`/`kind`, iterates `Spec.Allow`, maps each clause to a
  `closure.ScopeClause` via the constructors, runs `Validate()`, and returns a **`ScopePredicate`**.
  An unknown dim or invalid clause → wrapped error (fail-closed). The compiler is the one place that
  knows the dim→clause mapping, so adding a dimension later is local.
- **`ScopePredicate`** carries the compiled `[]closure.ScopeClause` plus `MaxSeverity` (DESIGN §6:
  reported via severity `B`, does *not* change the verdict — parsed and carried now, enforced later).
  `closure.Safe` is unchanged: callers pass `predicate.Clauses` to it. (Exact shape — named struct vs
  alias, and how clauses are exposed — is the design-doc's call; see Open questions.)
- **Loader** (`internal/scenario`) gains TaskContract YAML parsing (it already imports
  `sigs.k8s.io/yaml`), producing a `scope.TaskContract`, then calls `scope.Compile`. This keeps
  `scope/` YAML-free and embeddable. Existing `scope.yaml` parsing stays for back-compat.
- **`archguard`** extended to assert `scope/...` imports only stdlib + the `github.com/ridik-il/krsm/`
  module prefix.

## Steps (vertical slices)
1. [ ] **`scope/` package: `TaskContract` + `Compile` (resource + selector).** The struct, the
   compiler, fail-closed on unknown dim, `ScopePredicate` out. Unit tests (`package scope`): a
   TaskContract with a resource clause and a selector clause compiles to the expected
   `closure.ScopeClause`s; an unknown dim and a malformed clause each return an error; `maxSeverity`
   is carried. **Demo:** `go test ./scope`.
2. [ ] **End-to-end + guard.** Loader parses a `TaskContract` YAML → `scope.TaskContract` →
   `scope.Compile` → `closure.Safe`; prove a TaskContract YAML classifies the selector scenario (20)
   and the corpus. Extend `archguard` to `scope/`. **Demo:** `krsm check` (or a golden) over a
   TaskContract-driven input returns the right verdict; `go test ./internal/archguard` green.

## Risks & unknowns
- **`archguard` extension subtlety.** Must allow the intra-module `closure` import while still
  catching an external leak. Mitigation: allowlist the `github.com/ridik-il/krsm/` prefix; add a
  negative assertion idea (a deliberately-wrong import would fail) in review.
- **`ScopePredicate` shape churn.** If it forces a `Safe` signature change, that's a second
  back-to-back public break. Mitigation: keep `Safe([]ScopeClause)`; `ScopePredicate` *exposes*
  clauses, it doesn't replace the param.
- **Corpus wiring / back-compat.** Introducing TaskContract YAML must not break the 20 existing
  `scope.yaml` goldens. Mitigation: additive — loader supports both; the existing path is untouched.
- **YAML schema fidelity.** The corpus TaskContract must match DESIGN §6 (`krsm.io/v1`,
  `kind: TaskContract`, `spec.allow`, `maxSeverity`) so it is the real contract, not a toy.

## Definition of done
- `scope.Compile(TaskContract)` returns a `ScopePredicate` for resource + selector dims; unknown dim
  → compile error.
- A **`TaskContract` YAML** loads, compiles, and **correctly classifies the corpus including
  scenario 20** (ROADMAP v0.3 done-when).
- `scope/` is stdlib-only — `archguard` extended to cover it and green.
- All 20 existing golden scenarios stay green; `closure.Safe` signature unchanged.
- `make check` green; `CHANGELOG.md` entry; an ADR records the TaskContract/compiler decision
  (likely ADR-0009; or a note if it's purely realising ADR-0003).

## Open questions
1. **`ScopePredicate` shape:** named struct `{Clauses []closure.ScopeClause; MaxSeverity …}` vs a bare
   `[]closure.ScopeClause` alias. (Leaning: a struct, so `maxSeverity` and future per-predicate
   metadata have a home; `Clauses` field handed to `Safe`.)
2. **How a TaskContract enters the corpus:** an optional `taskcontract.yaml` per scenario (loader
   prefers it over `scope.yaml` when present), one standalone example compiled in a `scope`/loader
   test, or convert a scenario's `scope.yaml`. (Leaning: optional `taskcontract.yaml` on scenario 20,
   loader falls back to `scope.yaml` — additive, keeps all 20 goldens green.)
3. **`maxSeverity`:** parse + carry now (no enforcement; severity `B` is a later concern) or omit the
   field entirely this slice? (Leaning: parse + carry, no enforcement — it's in the DESIGN §6 schema.)
4. **`apiVersion`/`kind` strictness:** reject a wrong `apiVersion`/`kind` at compile, or ignore? (For a
   security control: reject — fail-closed on an unrecognised contract.)
5. **ADR:** new ADR-0009 (TaskContract compiler + scope/ package) vs a note on ADR-0003. (Leaning: a
   short ADR-0009, since the package boundary and ScopePredicate shape are real decisions.)

## Future v0.3 slices (decomposition — out of scope here)
