# Plan: v0.4-live-cluster-reads
Status: draft ¬∑ Date: 2026-06-23 ¬∑ Slug: `v0.4-live-cluster-reads`

## Problem
KRSM can compute a correct closure and a derived default scope, but only against a
hand-written `cluster.yaml` snapshot. The motivating milestone (DESIGN ¬ß2, ROADMAP v0.4)
is to run that same verdict against a **real cluster**: read live state read-only, build
the four relations from it, and have `krsm check` snapshot a `kind` cluster and compute
closure for a given action with **no manual `cluster.yaml`**. ADR-0011's consequence is
explicit: "every level still needs v0.4 (live reads) ‚Ä¶ nothing works offline." This is
the slice that makes the derived-scope work (already merged) usable on a cluster.

The other half of v0.4 ‚Äî the **derived default scope** (`ownership`/`namespace`
dimensions, `scope.Derive`, audit/enforce mode, goldens 23‚Äì26) ‚Äî is **done and merged**
(`docs/plans/v0.4-derived-scope.md`, Status: done). That plan recorded its own non-goal:
"live client-go informers / `kind` snapshotting (rest of v0.4, **separate plan**)." This
is that separate plan.

## Goals / Non-goals
- **Goal:** a new package outside `closure`/`scope` (proposed `internal/cluster`) that
  reads a live cluster via client-go and implements the existing **`closure.State`
  interface** (`closure/state.go`) ‚Äî a second `State` implementation beside
  `NewScanState`, *not* an engine change.
- **Goal:** that provider extracts the same four relations (DESIGN ¬ß3) from **unstructured**
  objects across **arbitrary GVKs** reached via discovery/RESTMapper ‚Äî ownerReference
  cascade (real `metadata.uid`/`ownerReferences`), namespace containment (real
  namespaced-vs-cluster-scoped from discovery, retiring the static `clusterScopedKinds`
  map), label-selector binding, and cross-refs (env/envFrom/volumes/imagePullSecrets/
  finalizers/scaleTargetRef).
- **Goal:** a CLI path so `krsm check` can target a live cluster ‚Äî the action target is
  specified on the command line (no `request.yaml`), and kubeconfig/context/namespace are
  resolved from standard client-go config plus flags.
- **Goal:** **fail-closed** (DESIGN ¬ß5) when the cluster read is partial or errors ‚Äî an
  unresolvable target or an unreadable kind must deny, never silently shrink the closure.
- **Goal:** keep `closure` and `scope` **stdlib-only** (`internal/archguard`); client-go
  enters `go.mod` for the first time but stays out of the embeddable SDK.
- **Goal:** address the supply-chain surface of the first k8s deps (version pinning,
  `govulncheck`/`dependency-review`/CodeQL already in CI) in the plan.
- **Goal:** the ROADMAP "Done when" proven on a `kind` cluster ‚Äî correct closure for a
  real action with no manual snapshot, **and** a derived-default escape flagged with no
  contract written.
- **Non-goal:** the admission webhook, `namespaceSelector`/`objectSelector` request
  scoping, enforce-as-deny-pre-persistence (v0.5).
- **Non-goal:** the `krsm.io/target` annotation (Level 1), templates (Level 2), the
  `reference` scope dimension.
- **Non-goal:** any change to `closure.Safe`'s signature, the closure walk, or the
  derived-scope/mode work already merged.
- **Non-goal:** shared informers / a watch-fed incremental index. This slice is a
  **one-shot list-and-index** for a single `krsm check` (see decision 1); the index
  builder is shaped so the v0.5 webhook can later wrap it in informers without a rewrite.
- **Non-goal:** mutating the cluster. Reads only (`get`/`list`); no create/update/delete.

## Current state
- `closure/state.go` ‚Äî the `State` interface (`Get`, `OwnedChildren`, `NamespaceContents`,
  `PodsSelectedBy`, `PodsMatching`, `SelectorsTargeting`, `SelectorsMatchingLabels`,
  `Consumers`, `ControllersTargeting`) and `scanState`/`NewScanState`, the linear-scan
  in-memory provider. The live provider must satisfy this **same interface**.
- `closure/model.go` ‚Äî `GVK`/`Ref`/`Object`/`Action`/`OwnerRef`/`CrossRef`/`LabelSelector`.
  `Ref.UID` is "the authoritative match key"; `Ref.key()` is `uid:<uid>` when present, else
  the human `Kind/ns/name`. `Object` carries only the fields the four relations read.
- `closure/safe.go` ‚Äî `Safe(s State, a Action, scope []ScopeClause)`; fail-closed when
  `Closure` is empty (`safe.go:22`), i.e. the target is absent from state.
- `closure/closure.go` ‚Äî the BFS walk; uses only `State` methods, so a faithful live
  `State` plugs in unchanged.
- `internal/scenario/scenario.go` ‚Äî the YAML loader. Builds `closure.Object`s from fixed
  kinds; synthesizes uids via `uidOf(kind, ns, name)` (a *string*, not a real uid);
  resolves owners by the same synthetic uid; has a static `clusterScopedKinds` map
  (`scenario.go:249`) standing in for discovery "until v0.4 reads it live" ‚Äî this slice is
  that v0.4. The relation-extraction logic (`crossRefsFrom`, `selectorFrom`,
  `podSpecCrossRefs`) is the reference for what the live provider must reproduce from
  unstructured objects.
- `cmd/krsm/main.go` ‚Äî `runCheck` loads a scenario dir, runs `mode.Apply(closure.Safe(...))`,
  writes the report. The live path extends this command with a new invocation form.
- `internal/archguard/archguard_test.go` ‚Äî `TestClosureIsStdlibOnly` /
  `TestScopeIsStdlibOnly` fail if anything under `closure/...` or `scope/...` imports a
  non-stdlib, non-own-module path. The new package must sit **outside** both trees.
- `go.mod` ‚Äî only non-stdlib dep today is `sigs.k8s.io/yaml`. CI (`.github/workflows/ci.yml`)
  runs `govulncheck@v1.4.0` (pinned), `dependency-review` (fail-on high, deny copyleft),
  CodeQL, Scorecard, signed-commits. Adding client-go must keep all of these green.
- Goldens 01‚Äì26 stay the source of truth for *correctness*; the live provider is validated
  by re-deriving the same relations, not by replacing the corpus.

## Approach
Add a **second `State` implementation** that reads a live cluster, behind the existing
interface seam ‚Äî the v0.1 design comment in `state.go` already anticipates this ("a later
informer-backed indexed implementation satisfies the same interface without changing
callers"). The engine, `scope`, and the goldens are untouched.

New package **`internal/cluster`** (internal, so it never becomes part of the embeddable
SDK and never trips archguard; DESIGN ¬ß7 names a future public `state/` package, but
keeping it `internal/` for v0.4 avoids committing to a public client-go-shaped API before
the webhook exists). It has two responsibilities, split into two files/types so the v0.5
webhook can reuse the second half:

1. **A reader** ‚Äî discovery + RESTMapper + a `dynamic` client that lists the relevant GVKs
   from the cluster and returns `[]unstructured.Unstructured` plus the discovered
   namespaced/cluster-scoped scope of each GVK.
2. **An index builder** ‚Äî a pure function `unstructured + scope info ‚Üí []closure.Object`,
   then `closure.NewScanState(objs)` (reused as-is). This is the half the webhook will
   later feed from informer caches instead of a one-shot list; isolating it keeps the
   "build the four relations from unstructured" logic testable with **fake objects, no
   cluster** (table tests, the bulk of the TDD here).

The provider returns `closure.State` to the CLI exactly as `scenario.Load` returns one
today, so `runCheck`'s `closure.Safe` / `mode.Apply` / `writeReport` flow is shared
verbatim between the file-snapshot path and the live path.

### Decisions taken (the five questions)

1. **One-shot list, not informers ‚Äî for this slice.** A single `krsm check` needs a point-in-
   time snapshot, computes one closure, and exits; a shared-informer cache (warm-up, watch,
   resync, lifecycle) buys nothing for a one-shot CLI and is pure cost. Recommend **discovery
   ‚Üí `dynamic` list ‚Üí index ‚Üí `NewScanState`**, fail-closed on any list error. The index
   builder (step "build `[]closure.Object` from unstructured") is written as a standalone
   pure function so the v0.5 webhook can drive it from `SharedInformerFactory` caches with no
   rewrite ‚Äî informers are deferred, the **seam for them is not**. (Alternative ‚Äî build the
   shared-informer index now and have the CLI block on one sync ‚Äî rejected: it front-loads
   the webhook's hardest concern into a CLI slice and slows every `check`.)

2. **Relation extraction from unstructured.** Port the `scenario.go` extractors
   (`selectorFrom`, `crossRefsFrom`, `podSpecCrossRefs`, owner/finalizer reading) to read
   from `unstructured.Unstructured` via `NestedSlice`/`NestedMap`/`NestedString` instead of
   typed structs, keeping the *same field paths* (`spec.selector`, `spec.podSelector` for
   NetworkPolicy, `spec.template.spec.{volumes,containers,initContainers,...}`,
   `spec.scaleTargetRef`, `metadata.finalizers`, `metadata.ownerReferences`). Two things
   change versus the loader:
   - **Real uids.** `Ref.UID` is the live `metadata.uid`; `OwnerRef.UID` is the live
     `ownerReferences[].uid` (no `uidOf` synthesis, no name-based owner resolution ‚Äî the
     owner edge is already a real uid on the child, which is exactly what `OwnedChildren`
     matches on). This makes the live provider *more* faithful than the loader, not less.
   - **Cross-ref uids.** A volume/env reference names only a kind+name, not a uid; resolve
     each referenced (Kind, ns, name) to a real uid via a name‚Üíuid index built over the
     listed objects (and leave UID empty when the referent was not listed ‚Äî `crossRefMatches`
     already falls back to Kind/ns/name, and a referent in an un-listed kind is handled by
     the fidelity boundary, decision 4). GVKs come from discovery/RESTMapper, not the
     hard-coded `GVK{Version:"v1",Kind:"ConfigMap"}` literals the loader uses.
   The static `clusterScopedKinds` map is **replaced by live discovery** for the live path
   (the loader keeps it ‚Äî YAML can't carry scope). The `internal/scenario` extractors and
   the `internal/cluster` extractors are deliberately *parallel, not shared* in this slice
   (one reads typed YAML structs, the other unstructured maps); a later refactor may extract
   a common relation-projection helper, but unifying them now is out of scope.

3. **CLI shape.** Add a live form that takes the action on the command line, e.g.
   `krsm check --context kind-krsm delete deployment/web -n prod`. The presence of cluster
   flags (`--context`/`--kubeconfig`, or an explicit `--cluster`) selects the live path; a
   positional directory still selects the existing scenario-dir path (unchanged). Target
   parsing reuses the same `Kind/name` (+ `-n namespace`) form the SCOPE line already prints
   (`Kind[.group]/ns/name`). Kubeconfig/context/namespace resolve through client-go's
   standard loading rules (`KUBECONFIG`, `~/.kube/config`, current-context) with flags
   overriding. `[assumed: the verb set is the existing closure.Verb values (delete/update/
   patch/scale/restart); mutation payloads (old/new) are not expressible on the CLI yet, so
   the live path supports the non-mutation verbs first and defers selector/finalizer-mutation
   actions ‚Äî which need a request payload ‚Äî to a follow-up.]`

4. **Fidelity boundary + fail-closed.** Default to a **whole-cluster** list of the GVKs the
   four relations can traverse (workloads, pods, services, configmaps, secrets, PVCs, PDBs,
   networkpolicies, HPAs, namespaces, plus discovered CRDs that participate via
   ownerReferences). Scope-narrowing (only-GVKs-reachable-from-the-target) is an
   **optimization deferred to the webhook**, where latency matters; a CLI one-shot can afford
   the broad list and the broad list is the *safe* default (a narrower list risks missing a
   closure member). Fail-closed rules (DESIGN ¬ß5), each a test:
   - **Unresolvable target** ‚Üí the existing `Safe` fail-closed deny (`Closure` empty) already
     covers it; ensure a target the cluster doesn't have produces it, with a clear reason.
   - **Unreadable/forbidden kind** (RBAC denies `list`, API group absent) ‚Üí **deny the whole
     check** with a fail-closed reason, never proceed on a partial snapshot that could hide a
     closure member. A partial read is an *unknown* closure, which DESIGN ¬ß5 says must deny.
   - **Discovery failure** ‚Üí fail-closed, do not fall back to the static kind map.

5. **`kind` acceptance test.** Prove "Done when" with a **build-tagged integration test**
   (`//go:build cluster`) plus a `make e2e` target that (a) creates a `kind` cluster, (b)
   applies a fixture mirroring scenario `01` (the intra-namespace cascade) and an in-scope
   action, (c) runs the real `krsm` binary against it, asserting the derived-default escape
   is flagged and the in-scope action allowed. The default `make check` / CI `go test` does
   **not** run it (no `kind` in CI yet ‚Äî `[assumed: the CI runner has no kind/Docker
   preconfigured; adding a kind step to ci.yml is a follow-up]`). It is a **local-and-manual
   gate** for v0.4, wired so a later CI job (`kind-action`) can flip it on. This keeps the
   unit suite hermetic (the index builder is fully tested with fake unstructured objects)
   while the one `kind` test anchors the milestone's observable claim.

### Dependency / supply-chain position
- Add the **minimum** client-go surface: `k8s.io/client-go` (dynamic + discovery +
  clientcmd), `k8s.io/apimachinery` (unstructured, RESTMapper, schema), `k8s.io/api` only if
  a typed decode is needed (prefer unstructured to avoid it). Pin to a single coherent
  client-go minor that supports the project's Go line; let Dependabot track bumps.
- These live **only** under `internal/cluster` and `cmd/krsm`; `archguard` already fails the
  build if they leak into `closure`/`scope`. Add an archguard-style assertion that the live
  k8s deps do **not** appear in `go list -deps ./closure/... ./scope/...` (the existing two
  tests already enforce this; confirm they still pass after the dep is added ‚Äî that *is* the
  guard).
- CI: `govulncheck ./...` now scans a much larger dep tree ‚Äî expect it to surface advisories;
  resolve by bumping the pinned client-go, not by suppressing. `dependency-review` denies
  GPL/AGPL ‚Äî client-go is Apache-2.0, fine. Note the larger attack surface in the PR.

## Steps (vertical slices)
Each slice ends in something observable; the early ones need no live cluster (fake
unstructured), the cluster-touching behavior is proven last.

1. [ ] **Index builder over fake unstructured (no cluster, no deps-in-engine).** Add
   `internal/cluster` with `BuildObjects([]unstructured.Unstructured, scopeInfo) ([]closure.Object, error)`
   plus a `name‚Üíuid` resolver, porting the `scenario.go` relation extractors to unstructured.
   Table tests with hand-built unstructured objects covering all four relations (owner edge
   by real uid, namespace containment, selector incl. NetworkPolicy `podSelector` and
   `matchExpressions`, cross-refs incl. `spec.template.spec` and projected/init/ephemeral/
   imagePullSecret/scaleTargetRef/finalizer). *Demo: feed fixtures equivalent to scenario
   `01`, get the same `[]closure.Object`; assert closure over `NewScanState` of them matches
   the golden's escaping set.*

2. [ ] **Live reader ‚Üí indexed `closure.State` (touches a cluster, behind a build tag for
   the live bits).** Add the discovery + RESTMapper + `dynamic` list, returning the
   namespaced/cluster-scoped scope per GVK, then `BuildObjects` ‚Üí `NewScanState`. Unit-test
   the discovery‚Üíscope mapping and the GVK selection with a fake discovery client; the actual
   list is exercised in step 5. Fail-closed on list/discovery error. *Demo (unit): a fake
   discovery+dynamic set lists N objects and the provider returns a `State` whose
   `NamespaceContents`/`OwnedChildren` counts match ‚Äî print the indexed `Ref` count.*

3. [ ] **CLI live path: target-on-command-line + kubeconfig resolution.** Extend `runCheck`
   to accept `krsm check [--context X] [--kubeconfig P] <verb> <Kind/name> [-n ns]`, select
   the live path when cluster flags are present, parse the target, build the live `State`, and
   run the **shared** `closure.Safe`/`mode.Apply`/`writeReport`. Scenario-dir mode unchanged.
   Test target parsing and path-selection in isolation (no cluster). *Demo: `krsm check
   --context ‚Ä¶ delete deployment/web -n prod` reaches `Safe` and prints ACTION/SCOPE/CLOSURE/
   VERDICT against a faked state; the existing `krsm check <dir>` still works.*

4. [ ] **Fail-closed on partial/erroring reads.** Wire the deny-on-unreadable-kind and
   deny-on-discovery-failure paths into the live provider with distinct reasons; ensure an
   unresolvable target produces the existing `Safe` fail-closed deny. Table tests with a fake
   client returning a forbidden/absent kind. *Demo: a forbidden `list` on one kind makes
   `krsm check` exit non-zero with a fail-closed reason, not a smaller closure.*

5. [ ] **`kind` acceptance test + `make e2e` (proves the ROADMAP "Done when").** A
   `//go:build cluster` test (or a script driving the built binary) that stands up `kind`,
   applies a scenario-`01`-equivalent fixture and an in-scope fixture, runs `krsm check`
   live, and asserts: derived-default escape **flagged**, in-scope action **allowed**, both
   with no `cluster.yaml` and no contract. Add `make e2e`; document it as a local gate; leave
   a commented CI job stub. *Demo: on a real `kind` cluster, `make e2e` is green and prints
   the flagged escaping resources.*

Each slice goes through `design-check-tdd` ‚Üí `tdd`; `make check` green after each (steps
1‚Äì4 hermetic; step 5 gated behind `make e2e`).

## Risks & unknowns
- **Relation parity between the YAML loader and the live provider (highest).** The live
  extractor must reproduce `scenario.go`'s four relations exactly, or the live verdict
  diverges from the golden. Mitigation: step 1 asserts the live `[]closure.Object` produces
  the *same closure/escaping set* as the golden corpus for equivalent fixtures, before any
  cluster is touched ‚Äî the corpus becomes the parity oracle.
- **Discovery/RESTMapper breadth.** Which GVKs to list, and correctly classifying
  cluster-scoped vs namespaced for CRDs, is fiddly; an over-narrow list silently shrinks the
  closure (a false negative ‚Äî the one error class a safety gate must not have). Mitigation:
  default broad, fail-closed on any kind that can't be read, and carry the `clusterScopedKinds`
  safety-invariant comment's spirit (under-scoping is unsafe; over-inclusion is merely
  conservative).
- **client-go version / Go-line compatibility.** client-go pins a minimum Go; CI builds on
  `go.mod`'s line (currently 1.24). A client-go that needs newer Go forces a `go.mod` bump.
  Mitigation: pick a client-go minor compatible with the stated Go line, or bump the line
  deliberately and note it.
- **govulncheck noise from the new tree.** A large dep tree often surfaces advisories.
  Mitigation: resolve by bumping the pinned client-go, never by suppressing; budget time for
  it in the PR.
- **No `kind` in CI.** The headline acceptance test can't run in the current CI. Mitigation:
  local-and-manual gate now, CI job stub for later; the hermetic index-builder tests carry
  the correctness load so the milestone isn't gated solely on a manual run.
- **CLI mutation actions.** selector/finalizer-mutation verdicts need an old/new payload that
  has no CLI form yet. Mitigation: ship non-mutation verbs first (decision 3), defer payload
  actions to a follow-up; record as an open question.

## Definition of done
Tied to ROADMAP v0.4 "Done when":
- On a `kind` cluster, `krsm check` reports the **correct closure for a real action with no
  manual `cluster.yaml`** (live discovery + list + index ‚Üí `Safe`), and **flags an escaping
  action under the derived default scope with no contract written** (reusing the merged
  `scope.Derive` + audit mode). Proven by `make e2e` / the `//go:build cluster` test.
- A partial or erroring cluster read **fails closed** (deny with a distinct reason), never
  admitting a shrunk closure.
- `closure` and `scope` remain **stdlib-only** (`internal/archguard` green); client-go lives
  only under `internal/cluster` + `cmd/krsm`.
- `closure.Safe`'s signature, the closure walk, and goldens 01‚Äì26 are **unchanged**.
- `make check` green (`gofmt`, `go vet`, `golangci-lint`, `staticcheck`, `go test -race`);
  CI green including `govulncheck` and `dependency-review` over the new deps.

## Open questions
1. Package name/location: `internal/cluster` (this plan's lean) vs the DESIGN ¬ß7-named public
   `state/`. Lean `internal/` for v0.4 ‚Äî defer a public client-go API until the webhook
   exists. (Design-doc to confirm.)
2. Typed vs unstructured decode: stay fully unstructured (this plan's lean, avoids `k8s.io/api`
   and keeps the extractor generic over CRDs) vs typed clients for the well-known kinds. Lean
   unstructured.
3. CLI surface: implicit path-selection by flag presence (this plan) vs an explicit
   `--cluster` / subcommand split (`krsm check-cluster`). Lean implicit, but a `--cluster`
   bool is a cheap disambiguator if parsing is ambiguous.
4. How to express a **mutation** action (selector/finalizer change) on the CLI ‚Äî needs an
   old/new payload. Deferred; non-mutation verbs first. Does the live path need it for v0.4's
   "Done when" (which is a `delete` cascade)? `[assumed: no ‚Äî the flagship is a cascading
   delete; mutations are a follow-up.]`
5. `kind` in CI: add a `kind`-action job now (gated/optional) or defer until v0.5 (which needs
   `kind` for the webhook e2e anyway)? Lean: defer, leave the job stub.
6. client-go version to pin and whether it forces a `go.mod` Go-line bump from 1.24.
   (Spike in step 2.)
